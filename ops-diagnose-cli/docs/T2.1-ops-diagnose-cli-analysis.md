# T2.1 算子编译诊断 CLI 分析报告

## 1. 任务目标

T2.1 的目标是实现 `triton-anchor-diagnose` 命令行工具，用于定位 Triton Anchor 算子编译失败的精确位置。

本任务包含三项交付：

1. 实现 `PassDiagnostic` 类。
2. 实现 CLI 入口 `triton-anchor-diagnose`。
3. 注册为 `console_scripts` entry point。

最终实现还额外接入了：

1. `triton-linalg` adapter pipeline 诊断。
2. MLIR op/location 的 best-effort 解析。
3. `--python MODULE:FUNCTION` 运行时生成 TTIR 的无预 dump 诊断模式。
4. Sophgo JIT 编译失败自动诊断 hook，由 `TRITON_ANCHOR_DIAGNOSE_ON_ERROR=1` 控制。

该功能的核心价值是把当前“整个 PassManager 失败”的粗粒度错误，拆解为“具体哪个 pass 失败、失败前 IR 是什么、失败 op/location 在哪里”的诊断结果。

## 2. 当前仓库现状

当前仓库已有编译管线相关实现：

- `python/triton_anchor/pipeline.py`
  - 提供 `build_ttir_pipeline(pm, hw=None)`。
  - 一次性向同一个 MLIR `PassManager` 添加 7 个 mandatory TTIR pass。
  - 提供 `make_ttir(mod, metadata, hw=None)`，内部执行完整 TTIR pipeline。

- `python/triton_anchor/adapters/triton_linalg_adapter.py`
  - 提供 Triton 到 Linalg 的 adapter。
  - 内部 `_add_passes(pm, passes)` 一次性添加 linalg conversion pass pipeline。

- `setup.py`
  - 当前实际安装入口主要由 `setup.py` 管理。
  - 已有 `entry_points={"triton.adapters": [...]}`。
  - 尚未注册 `console_scripts`。

- `/workspace/triton-sophgo-backend/triton_sophgo/compiler.py`
  - Sophgo JIT 后端实际执行 `fn[grid](...)` 编译链路。
  - 编译阶段为 `TTIR -> Linalg -> PPL IR -> .so`。
  - `_make_ttir()` 和 `_make_linalg()` 之前是一次性运行完整 `PassManager`，失败时不会自动调用 `PassDiagnostic`。

当前问题是：`PassManager` 一次性运行完整 pipeline 时，如果中间某个 pass 抛异常，用户通常只能看到最终异常，难以判断是哪个 pass 导致失败，也不方便获取失败前 IR 快照。

## 3. 设计原则

### 3.1 逐 pass 运行

诊断模式不应直接复用完整 `build_ttir_pipeline(pm)` 的一次性执行方式，而应将 pipeline 拆成可枚举的 pass descriptor：

```python
("common.inliner", passes.common.add_inliner)
("ttir.combine", passes.ttir.add_combine)
("common.canonicalizer", passes.common.add_canonicalizer)
...
```

每次只创建一个新的 `PassManager`，只添加当前 pass，然后执行：

```python
pm = ir.pass_manager(mod.context)
add_pass(pm)
pm.run(mod)
```

这样可以准确定位首个失败 pass。

### 3.2 保留 IR 快照

每个 pass 执行前后都应保存 IR：

```text
01-common.inliner.before.mlir
01-common.inliner.after.mlir
02-ttir.combine.before.mlir
02-ttir.combine.after.mlir
...
```

当 pass 失败时，至少需要保存：

- 当前 pass 执行前 IR。
- 错误信息。
- Python traceback，可选。
- 诊断 summary。

### 3.3 首个失败点优先

默认诊断模式应在首个失败 pass 停止，并返回非 0 exit code。这样符合编译失败诊断的主要诉求：定位最早破坏编译的 pass。

后续可以扩展 `--continue-on-error`，但首版不建议增加复杂度。

### 3.4 自动诊断不改变默认编译行为

Sophgo JIT hook 必须由环境变量显式打开：

```bash
TRITON_ANCHOR_DIAGNOSE_ON_ERROR=1
```

默认关闭时不应额外序列化 IR，也不应改变编译输出和性能。开启后，只在 `_make_ttir()` 或 `_make_linalg()` 抛异常时复跑对应 pipeline 做诊断。

## 4. PassDiagnostic 类设计

建议新增文件：

```text
python/triton_anchor/diagnostics.py
```

### 4.1 数据结构

建议定义诊断结果结构：

```python
from dataclasses import dataclass
from pathlib import Path


@dataclass
class PassDiagnosticResult:
    ok: bool
    pipeline: str
    failed_pass: str | None
    failed_index: int | None
    total_passes: int
    output_dir: Path
    error: str | None = None
    before_ir: Path | None = None
    after_ir: Path | None = None
```

也可以进一步细化每个 pass 的执行记录：

```python
@dataclass
class PassRunRecord:
    index: int
    name: str
    ok: bool
    before_ir: Path
    after_ir: Path | None
    error: str | None = None
```

### 4.2 PassDiagnostic API

建议首版提供以下 API：

```python
class PassDiagnostic:
    def __init__(self, output_dir=None, enable_debug=True, save_success_snapshots=True):
        ...

    def diagnose_ttir(self, mod, hw=None) -> PassDiagnosticResult:
        ...
```

其中：

- `output_dir`：保存 IR 快照和 summary 的目录。
- `enable_debug`：是否调用 `pm.enable_debug()`。
- `save_success_snapshots`：是否保存成功 pass 的 after IR。默认保存，便于追踪 IR 演化。

### 4.3 TTIR pipeline pass 列表

首版应覆盖 `pipeline.py` 里的 7 个 mandatory pass：

```python
passes.common.add_inliner(pm)
passes.ttir.add_combine(pm)
passes.common.add_canonicalizer(pm)
passes.ttir.add_reorder_broadcast(pm)
passes.common.add_cse(pm)
passes.common.add_licm(pm)
passes.common.add_symbol_dce(pm)
```

如果传入 `hw`，可以按现有 `build_ttir_pipeline()` 逻辑追加条件 pass：

- GPGPU 路径：`passes.ttir.add_rewrite_tensor_pointer`
- 可选 loop unroll：`passes.ttir.add_loop_unroll`
- 可选 expression restructuring：`passes.ttir.add_expression_restructing`

首版已实现 mandatory pass 诊断，并保留基于 `HWCapability` 追加条件 pass 的能力。

### 4.4 triton-linalg pipeline pass 列表

`triton-linalg` adapter 诊断复用同一套 `PassDiagnostic` 框架，将 `TritonLinalgAdapter._add_passes()` 拆成 pass descriptor：

```text
triton_linalg.wrap_func_body_with_single_block.pre
common.inliner
common.canonicalizer.pre
triton_linalg.canonicalize_triton
triton_linalg.pointer_strength_reduction
common.canonicalizer.after_pointer_strength_reduction
triton_linalg.triton_to_linalg
triton_linalg.extract_like_move_backward
common.canonicalizer.post_conversion
triton_linalg.arith_to_linalg
triton_linalg.math_to_linalg
common.cse
common.licm
triton_linalg.wrap_func_body_with_single_block.final
```

这使得 `TTIR -> Linalg` lowering 失败时可以定位到具体 adapter pass。

## 5. CLI 设计

建议新增文件：

```text
python/triton_anchor/diagnose.py
```

### 5.1 命令格式

首版推荐支持 TTIR 文件输入：

```bash
triton-anchor-diagnose input.ttir --pipeline ttir --output-dir /tmp/anchor-diagnose
```

同时支持 Python kernel 运行时输入，用于“不预先 dump IR 文件”的诊断场景：

```bash
triton-anchor-diagnose \
  --python tests.test_smoke:_smoke_add_kernel \
  --signature '*fp32,*fp32,*fp32,i32' \
  --constant BLOCK=256 \
  --pipeline ttir \
  --output-dir /tmp/anchor-diagnose
```

也支持模块方式调用：

```bash
python -m triton_anchor.diagnose input.ttir --pipeline ttir
python -m triton_anchor.diagnose --python tests.test_smoke:_smoke_add_kernel --signature '*fp32,*fp32,*fp32,i32' --constant BLOCK=256
```

### 5.2 参数设计

建议参数：

```text
positional:
  input                     输入 IR 文件，支持 .ttir/.mlir；使用 --python 时可省略

options:
  --pipeline ttir           诊断 pipeline，首版支持 ttir
  --python MODULE:FUNCTION  从 Python @triton.jit 函数运行时生成 TTIR 并诊断
  --signature TYPES         Python kernel 参数类型，例如 '*fp32,*fp32,*fp32,i32'
  --constant KEY=VALUE      Python kernel 编译期常量，可重复，例如 BLOCK=256 或 4=256
  --output-dir DIR          诊断产物目录
  --no-debug                不调用 pm.enable_debug()
  --quiet                   只输出最终结果
  --dump-summary-json       额外输出 summary.json
```

首版最小参数集可以只实现：

```text
input
--python
--signature
--constant
--pipeline
--output-dir
--no-debug
```

`--python` 模式只支持 `--pipeline ttir`。它在 CLI 进程内导入 Python kernel，构造 `ASTSource`，生成内存中的 TTIR module，然后直接调用 `PassDiagnostic.diagnose_ttir(mod)`；不要求用户提前 dump `.ttir` 文件。

### 5.3 CLI 输出

成功时：

```text
OK: pipeline ttir completed, 7 passes executed.
diagnostic output: /tmp/anchor-diagnose
```

失败时：

```text
FAILED: pipeline ttir failed at pass 4/7: ttir.reorder_broadcast
before IR: /tmp/anchor-diagnose/04-ttir.reorder_broadcast.before.mlir
diagnostic detail: /tmp/anchor-diagnose/04-ttir.reorder_broadcast.diagnostic.txt
location: file:line:column
operation: tt.load
error: <exception message>
diagnostic output: /tmp/anchor-diagnose
```

exit code：

- 成功：`0`
- 编译/pass 失败：`1`
- CLI 参数错误或输入文件错误：`2`

## 6. 输入 IR 加载流程

CLI 应使用 Triton Python 绑定加载 MLIR：

```python
from triton._C.libtriton import ir, anchor

ctx = ir.context()
ir.load_dialects(ctx)
anchor.load_dialects(ctx)
mod = ir.parse_mlir_module(str(input_path), ctx)
mod.context = ctx
```

注意事项：

1. `anchor.load_dialects(ctx)` 可能依赖 C++ 扩展构建产物。
2. 如果 `triton._C.libtriton` 未构建或不可导入，CLI 应输出明确错误，而不是裸 traceback。
3. 对于 `.ttir` 与 `.mlir` 后缀，首版可以都按 MLIR module 解析。

### 6.1 Python Kernel 输入流程

当用户使用 `--python MODULE:FUNCTION` 时，CLI 不读取 IR 文件，而是在运行时生成 TTIR：

```python
import importlib
import triton
from triton._C.libtriton import ir, anchor

module = importlib.import_module(module_name)
fn = getattr(module, function_name)

ctx = ir.context()
ir.load_dialects(ctx)
anchor.load_dialects(ctx)

src = triton.compiler.ASTSource(
    fn=fn,
    signature={0: "*fp32", 1: "*fp32", 2: "*fp32", 3: "i32"},
    constants={4: 256},
)
ttir_module = src.make_ir(options=minimal_options, codegen_fns=None, context=ctx)
PassDiagnostic(output_dir=...).diagnose_ttir(ttir_module)
```

`--constant` 支持两种 key：

- 参数名：`--constant BLOCK=256`
- 参数下标：`--constant 4=256`

参数名会通过 kernel 函数的 `__code__.co_varnames` 映射到参数下标。

## 7. Sophgo JIT 失败自动诊断

除 CLI 外，最终实现还在 Sophgo 后端 JIT 编译链路中加入可选 on-error hook：

```text
/workspace/triton-sophgo-backend/triton_sophgo/compiler.py
```

开关：

```bash
TRITON_ANCHOR_DIAGNOSE_ON_ERROR=1
```

输出目录：

1. 设置 `TRITON_ANCHOR_DIAGNOSE_DIR` 时，写入该目录。
2. 否则设置了 `TRITON_DUMP_DIR` 时，写入 `<TRITON_DUMP_DIR>/<kernel>/anchor-diagnose/<pipeline>/`。
3. 否则写入当前目录下 `triton-anchor-diagnose/<kernel>/<pipeline>/`。

适用范围：

- 不只针对 `tests/test_jit.py` 中已有算子。
- 任意新增算子只要通过 Sophgo 后端 `fn[grid](...)` 触发编译，并在 `_make_ttir()` 或 `_make_linalg()` 阶段失败，都会自动进入同一套诊断逻辑。

输出示例：

```text
[AnchorDiagnose] pipeline triton-linalg failed at pass 7/14: triton_linalg.triton_to_linalg
[AnchorDiagnose] before IR: ...
[AnchorDiagnose] diagnostic detail: ...
[AnchorDiagnose] location: file:line:column
[AnchorDiagnose] operation: tt.elementwise_inline_asm
[AnchorDiagnose] summary: ...
```

## 11. 风险与边界

### 11.1 pass 拆分与原 pipeline 行为一致性

逐 pass 运行时，每个 pass 使用独立 `PassManager`。一般情况下这与同一个 `PassManager` 顺序执行等价，但如果 MLIR pass manager 存在跨 pass 状态，可能有细微差异。

建议将诊断模式定位为 debug 工具，不替代正式编译路径。

### 11.2 C++ 扩展依赖

CLI 需要 `triton._C.libtriton` 可用。未构建 C++ 扩展时，CLI 只能输出环境错误，不能执行 pass 诊断。

### 11.3 op/location 定位边界

当前已实现 MLIR op/location 的 best-effort 解析，主要依赖 pass 抛出的 exception 和 C++/MLIR 写到 stderr 的 diagnostic。若底层 pass 没有输出 location 或 op 名称，则只能稳定定位到 pass 和失败前 IR。

### 11.4 自动 hook 覆盖范围

Sophgo JIT 自动诊断当前覆盖 `_make_ttir()` 和 `_make_linalg()` 阶段。若失败发生在 Python 前端生成 TTIR 之前，或后续 PPLIR / `ppl-compile` 阶段，需要继续接入对应阶段的诊断逻辑。

## 13. 结论

T2.1 的最佳实现路径是新增独立诊断模块，并在正式 JIT 编译链路中通过环境变量开关接入 on-error hook。`PassDiagnostic` 通过逐 pass 执行和 IR 快照保存，能够稳定定位首个失败 pass；`triton-anchor-diagnose` CLI 负责把该能力暴露给用户；Sophgo JIT hook 负责让 `tests/test_jit.py` 以及后续新增算子的编译失败自动输出 pass/op/location 诊断。

当前版本已覆盖 TTIR、`triton-linalg` adapter、Python kernel 运行时输入和 Sophgo JIT on-error 诊断。后续如需覆盖 PPLIR / `ppl-compile`，可继续在 Sophgo 后端对应阶段接入同样的诊断框架。
