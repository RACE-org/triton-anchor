# T2.1 算子编译诊断 CLI 实现与测试结果

## 1. 实现概述

本次已完成首版 `triton-anchor-diagnose` 诊断 CLI，实现目标是将编译失败定位到具体 pass。

当前范围：

- 支持 Python/Triton 前端生成 TTIR 前失败的 pre-pass 诊断，阶段名为 `python-frontend`。
- 支持输入 `.ttir/.mlir` 在 MLIR parse 阶段失败的 pre-pass 诊断，阶段名为 `input-parse`。
- 支持 TTIR pipeline 逐 pass 诊断。
- 支持 triton-linalg adapter pipeline 逐 pass 诊断。
- 在每个 pass 执行前保存 IR 快照。
- 成功 pass 可保存 after IR 快照。
- 首个 pass 失败时停止，并报告失败 pass 名称、序号、错误信息和 before IR 路径。
- 尝试从 MLIR diagnostic / exception / 底层 stderr 中提取 MLIR location 与 op 名称。
- 失败时写入独立 `.diagnostic.txt`，保存错误、捕获到的底层诊断、location/op 和 traceback。
- 输出 `summary.json`，方便后续自动化分析。
- 提供 `python -m triton_anchor.diagnose` 和安装后的 `triton-anchor-diagnose` 命令入口。
- 支持 `--python MODULE:FUNCTION` 模式，在 CLI 运行时直接从 Python `@triton.jit` kernel 生成内存 TTIR 后诊断，不要求预先 dump `.ttir` 文件。
- 如果 `--python` 模式还没生成 TTIR 就失败，会写入 `python-frontend.diagnostic.txt` 和 `summary.json`。
- 如果文件输入在 `ir.parse_mlir_module()` 阶段失败，会写入 `input-parse.diagnostic.txt` 和 `summary.json`。
- 支持 Sophgo JIT 编译失败时按环境变量自动触发诊断；任意通过 `fn[grid](...)` 进入 Sophgo 后端的新旧算子都适用。

## 3. 核心实现

### 3.1 `PassDiagnostic`

实现位置：

```text
python/triton_anchor/diagnostics.py
```

新增主要类型：

- `PassDescriptor`
- `PassRunRecord`
- `PassDiagnosticResult`
- `PassDiagnostic`

核心行为：

1. 构造 TTIR pass descriptor 列表。
2. 每次创建新的 MLIR `PassManager`。
3. 每次只向 `PassManager` 添加一个 pass。
4. 执行前写入 before IR。
5. pass 成功后可写入 after IR。
6. pass 失败时捕获 Python exception 和底层 stderr diagnostic。
7. 从诊断文本中提取 MLIR file/line/column/op，尽量映射回 before IR 行。
8. pass 失败时记录异常、traceback、失败 pass 名称、序号、location/op 和诊断详情文件。
9. 写入 `summary.json`。

- `ttir.rewrite_tensor_pointer`

### 3.2 诊断原理与源码对应

#### 3.2.1 pass 定位原理

原正常编译流程是将整条 pipeline 的多个 pass 一次性加入同一个 MLIR `PassManager`，然后执行一次 `pm.run(mod)`。这种方式失败时只能知道整个 pipeline 失败，不能稳定判断哪个 pass 首先失败。

诊断模式改为逐 pass 执行：

1. 将 pipeline 拆成 `PassDescriptor(name, add_to_pass_manager)` 列表。
2. 每轮只创建一个新的 `PassManager`。
3. 每轮只加入当前一个 pass。
4. 执行当前 pass 前保存 before IR。
5. 当前 pass 成功后继续下一个 pass。
6. 当前 pass 抛异常时立即停止，并记录当前 pass 名称和序号。

核心源码：

```text
python/triton_anchor/diagnostics.py
```

对应函数和逻辑：

```text
PassDiagnostic._diagnose_pipeline()
```

关键代码位置：

```text
for index, descriptor in enumerate(descriptors, start=1):
    before_ir_text = str(mod)
    before_path = self._write_ir(index, descriptor.name, "before", mod)
    pm = ir.pass_manager(mod.context)
    descriptor.add_to_pass_manager(pm)

    try:
        diagnostic_text = self._run_pass(pm, mod)
    except Exception:
        result.failed_pass = descriptor.name
        result.failed_index = index
        ...
        return result
```

因此，`descriptor.name` 就是失败 pass 名称，`index` 就是失败 pass 序号，`before_path` 是失败前 IR 快照。

TTIR pass descriptor 构造位置：

```text
python/triton_anchor/diagnostics.py::build_ttir_pass_descriptors()
```

`triton-linalg` pass descriptor 构造位置：

```text
python/triton_anchor/diagnostics.py::build_triton_linalg_pass_descriptors()
```

#### 3.2.2 op/location 定位原理

具体 op/location 不是通过修改 C++ pass 获得的，而是 best-effort 解析底层 MLIR/C++ diagnostic：

1. 执行单个 pass 时，临时捕获进程 `stderr`。
2. MLIR/C++ pass 报错通常会输出类似：

   ```text
   loc("kernel.py":14:8): error: failed to legalize operation 'tt.elementwise_inline_asm'
   ```

3. Python 侧从 exception 文本和 captured stderr 中解析：
   - `loc("file":line:column)`
   - `failed to legalize operation 'tt.xxx'`
4. 如果报错里没有直接给出 op，则尝试用 location 回查 before IR 中对应行，再从 IR 行提取 op。

捕获 stderr 的源码：

```text
python/triton_anchor/diagnostics.py::_run_with_stderr_capture()
```

op/location 解析入口：

```text
python/triton_anchor/diagnostics.py::_extract_mlir_location()
```

location 正则匹配：

```text
python/triton_anchor/diagnostics.py::_find_location_match()
```

op 正则匹配：

```text
python/triton_anchor/diagnostics.py::_extract_operation_from_text()
```

用 location 回查 before IR：

```text
python/triton_anchor/diagnostics.py::_find_ir_line_for_location()
python/triton_anchor/diagnostics.py::_extract_operation_from_ir_line()
```

这个定位是 best-effort：如果底层 pass 只输出 `PassManager::run failed`，没有 location 或 op，工具仍能稳定定位失败 pass 和 before IR，但不一定能定位到具体 op。

### 3.3 MLIR op/location 定位

失败时会综合以下来源做 best-effort 解析：

- pass 抛出的 exception 文本。
- `pm.run()` 期间底层 C++/MLIR 写到 stderr 的 diagnostic。
- 当前 pass 执行前的 before IR 快照。

当前可识别的 location 形式包括：

```text
loc("file":line:column)
file.mlir:line:column
file.ttir:line:column
<stdin>:line:column
line:column
```

当前可识别的 op 形式包括：

```text
failed to legalize operation 'tt.load'
operation 'tt.store'
"tt.load"(...)
tt.store ...
```

定位结果会写入：

- CLI 输出。
- `summary.json`。
- `NN-pass-name.diagnostic.txt`。

### 3.4 pre-pass 诊断

新增能力覆盖“还没进入 pass”的两类失败：

```text
python/triton_anchor/diagnose.py
```

核心类型：

- `InputDiagnosticError`
- `InputDiagnosticResult`

#### 3.4.1 Python/Triton 前端失败

触发路径：

```text
python_triton kernel -> triton.compiler.ASTSource -> src.make_ir() -> TTIR
```

如果在导入 Python kernel、解析签名/constexpr、或 `src.make_ir()` 生成 TTIR 之前失败，CLI 不再只打印一行 `failed to prepare input module`，而是写入结构化诊断：

```text
stage: python-frontend
diagnostic detail: python-frontend.diagnostic.txt
summary: summary.json
```

实现位置：

```text
python/triton_anchor/diagnose.py::_make_ttir_from_python()
python/triton_anchor/diagnose.py::_write_input_diagnostic()
python/triton_anchor/diagnose.py::_print_input_diagnostic()
```

实现方式：

1. `triton` / `libtriton` 导入成功后，执行 Python kernel 加载和 TTIR 生成。
2. 对 `src.make_ir()` 增加 stderr 捕获，保留底层 frontend/MLIR diagnostic。
3. 失败时抛出 `InputDiagnosticError(stage="python-frontend")`。
4. CLI 捕获该异常后写 `python-frontend.diagnostic.txt` 和 `summary.json`。
5. 复用 `diagnostics.py::extract_mlir_location()` 从错误文本中 best-effort 提取 `file:line:column` 和 op。

该阶段还没有生成合法 TTIR module，所以不会有 failed pass，也不会有 `*.before.mlir`。诊断结果定位的是失败阶段和 frontend diagnostic。

#### 3.4.2 MLIR parse 失败

触发路径：

```text
input .ttir/.mlir -> ir.parse_mlir_module()
```

如果输入文件自身非法，`ir.parse_mlir_module()` 阶段失败，CLI 会写入：

```text
stage: input-parse
diagnostic detail: input-parse.diagnostic.txt
summary: summary.json
```

实现位置：

```text
python/triton_anchor/diagnose.py::_load_mlir_module()
python/triton_anchor/diagnose.py::_capture_stderr()
python/triton_anchor/diagnose.py::_write_input_diagnostic()
```

实现方式：

1. 文件存在性检查仍然保留，文件不存在直接返回参数错误。
2. `ir.parse_mlir_module()` 通过 `_capture_stderr()` 执行，捕获底层 MLIR parse diagnostic。
3. parse 失败时抛出 `InputDiagnosticError(stage="input-parse")`。
4. CLI 写 `input-parse.diagnostic.txt` 和 `summary.json`。
5. 如果 diagnostic 中包含 `file:line:column`，会读取原始输入文件，尽量回查对应 IR 行和 op。

该阶段同样还没有进入 pass，因此不会报告 failed pass。诊断目标是确认失败发生在输入 parse，而不是后续 TTIR / `triton-linalg` pass。

### 3.5 CLI 入口

实现位置：

```text
python/triton_anchor/diagnose.py
```

支持调用方式：

```bash
python -m triton_anchor.diagnose input.ttir --pipeline ttir
python -m triton_anchor.diagnose input.ttir --pipeline triton-linalg
python -m triton_anchor.diagnose --python tests.test_smoke:_smoke_add_kernel --signature '*fp32,*fp32,*fp32,i32' --constant BLOCK=256
triton-anchor-diagnose input.ttir --pipeline ttir
triton-anchor-diagnose input.ttir --pipeline triton-linalg
triton-anchor-diagnose --python tests.test_smoke:_smoke_add_kernel --signature '*fp32,*fp32,*fp32,i32' --constant BLOCK=256
```

CLI 参数：

```text
input
--python MODULE:FUNCTION
--signature SIGNATURE
--constant NAME_OR_INDEX=VALUE
--pipeline {ttir,triton-linalg}
--output-dir OUTPUT_DIR
--no-debug
--no-success-snapshots
--no-summary-json
--quiet
```

退出码约定：

- `0`：诊断 pipeline 成功。
- `1`：诊断过程中 pass 或运行逻辑失败。
- `2`：输入文件、参数或环境错误。

`--python` 模式约束：

- 只支持 `--pipeline ttir`。
- `--signature` 必填。
- `--constant` 可重复，key 可以是参数名或参数下标。
- 不需要预先存在 `.ttir` 文件；CLI 内部生成 TTIR module 后直接调用 `PassDiagnostic`。

失败输出会额外包含：

```text
diagnostic detail: /tmp/anchor-diag/NN-pass-name.diagnostic.txt
location: file:line:column
operation: tt.load
ir line: N: %0 = "tt.load"(...)
```

### 3.7 Sophgo JIT 失败自动诊断

实现位置：

```text
/workspace/triton-sophgo-backend/triton_sophgo/compiler.py
```

新增环境变量开关：

```bash
TRITON_ANCHOR_DIAGNOSE_ON_ERROR=1
```

默认不开启，不影响正常 JIT 编译性能和输出。开启后，任意通过 Sophgo 后端 `fn[grid](...)` 触发编译的算子，如果在 `_make_ttir()` 或 `_make_linalg()` 阶段失败，会自动调用 `PassDiagnostic` 复跑对应 pipeline，并输出：

```text
[AnchorDiagnose] pipeline ...
[AnchorDiagnose] before IR: ...
[AnchorDiagnose] diagnostic detail: ...
[AnchorDiagnose] location: file:line:column
[AnchorDiagnose] operation: ...
[AnchorDiagnose] summary: ...
```

诊断输出目录规则：

1. 如果设置 `TRITON_ANCHOR_DIAGNOSE_DIR`，写入该目录。
2. 否则如果设置 `TRITON_DUMP_DIR`，写入对应 kernel dump 目录下的 `anchor-diagnose/<pipeline>/`。
3. 否则写入当前工作目录下的 `triton-anchor-diagnose/<kernel>/<pipeline>/`。

该能力不是只针对 `tests/test_jit.py` 中已有四个算子；只要新增算子仍走 Sophgo JIT 后端的 TTIR 或 triton-linalg 编译阶段，失败时都会进入同一套诊断逻辑。

故意引入的问题是使用 `tl.inline_asm_elementwise`：

```python
@triton.jit
def bad_inline_asm_kernel(x_ptr, out_ptr, n, BLOCK: tl.constexpr):
    pid = tl.program_id(0)
    offs = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offs < n
    x = tl.load(x_ptr + offs, mask=mask)
    y = tl.inline_asm_elementwise(
        "mov.u32 $0, $1;",
        "=r,r",
        [x.to(tl.int32)],

        pack=1,
```

### 5.7 console script 注意事项

如果直接执行：

```bash
triton-anchor-diagnose input.ttir --pipeline ttir --output-dir /tmp/anchor-diag
```

出现：

```text
bash: triton-anchor-diagnose: command not found
```

原因不是 CLI 代码不可用，而是当前 `/opt/venv` 中安装的 `triton-anchor 0.2.0` 是本次修改前安装的旧包。`console_scripts` 命令是在包安装时生成到 `/opt/venv/bin/` 的；修改 `setup.py` / `pyproject.toml` 后，如果没有重新安装当前仓库，`/opt/venv/bin/triton-anchor-diagnose` 不会自动出现。

当前源码可以先用模块方式运行：

```bash
cd /workspace/triton-anchor
PYTHONDONTWRITEBYTECODE=1 PYTHONPATH=python /opt/venv/bin/python -m triton_anchor.diagnose input.ttir --pipeline ttir --output-dir /tmp/anchor-diag
```

若要生成 `triton-anchor-diagnose` 命令，需要在容器内重新安装当前仓库：

```bash
cd /workspace/triton-anchor
/opt/venv/bin/python -m pip install -e .
```

如果 editable install 失败，并出现类似：

```text
running build_ext
Traceback (most recent call last):
  File "/usr/local/bin/cmake", line 3, in <module>
    from cmake import cmake
ModuleNotFoundError: No module named 'cmake'
subprocess.CalledProcessError: Command '['cmake', '--version']' returned non-zero exit status 1.
```

说明当前容器里的 `/usr/local/bin/cmake` 是 Python wrapper：

```text
#!/usr/bin/python3.12
from cmake import cmake
```

但系统 Python / venv 中没有 `cmake` Python 模块。并且 `pip install -e .` 会触发本仓库 `setup.py` 中的 C++/CMake 扩展构建，所以仅为了生成 console script 也会先卡在 CMake 环境。

注意不要使用裸 `pip install -e .`。当前容器里 `/opt/venv/bin/` 可能只有 `pip3` / `pip3.12`，没有 `pip`；激活 venv 后裸 `pip` 可能解析到系统 pip。判断方式是看安装日志路径：如果出现 `/usr/local/lib/python3.12/dist-packages` 或 `/usr/bin/python3.12`，说明命令没有走 `/opt/venv`。建议始终使用：

```bash
/opt/venv/bin/python -m pip ...
```

完整安装应参考 `docs/build.md`，先准备构建依赖和 LLVM 环境：

```bash
cd /workspace/triton-anchor
source /opt/venv/bin/activate
source envsetup.sh
/opt/venv/bin/python -m pip install cmake ninja pybind11 wheel setuptools
/opt/venv/bin/python -m pip install -e . --no-build-isolation
```

如果只是临时验证 CLI 功能，而不想触发完整 C++ 构建，可以继续使用模块方式：

```bash
cd /workspace/triton-anchor
PYTHONDONTWRITEBYTECODE=1 PYTHONPATH=python /opt/venv/bin/python -m triton_anchor.diagnose --help
```

开发过程中如果新增了 CLI 参数，例如 `--python`，但 `triton-anchor-diagnose --help` 中还看不到，说明当前 `/opt/venv/bin/triton-anchor-diagnose` 仍来自旧 wheel。需要重新执行：

```bash
cd /workspace/triton-anchor
uv build --wheel --no-build-isolation
uv pip install dist/triton_anchor-*.whl
```

或者临时用源码模块方式验证：

```bash
PYTHONPATH=/workspace/triton-anchor/python:/workspace/triton-anchor python3 -m triton_anchor.diagnose --help
```

安装完成后确认：

```bash
which triton-anchor-diagnose
triton-anchor-diagnose --help
```

## 7. 后续建议

当前版本已完成 pass 级诊断、MLIR op/location 的 best-effort 定位、triton-linalg adapter pipeline 接入、Python kernel 运行时生成 TTIR 的内存诊断模式，以及 Sophgo JIT 编译失败自动诊断 hook。后续可继续增强：

1. 增加原生 MLIR diagnostic handler，减少对诊断文本正则解析的依赖。
2. 将 Sophgo JIT on-error hook 扩展到 PPLIR / `ppl-compile` 阶段。
3. 增加 `--run-ttir-before-linalg` 之类的串联诊断模式。
4. 支持更完整的 Triton 编译 options 参数，例如 `num_warps`、`num_stages`、`num_ctas`。
